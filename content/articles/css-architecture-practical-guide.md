# CSS 架构实战：从样式混乱到系统可复用

团队规模一旦增长，CSS 往往最先失控。最典型的问题是：

- 一个样式改动影响三处页面
- 同一种按钮有五种颜色
- 新人不敢删旧代码

根本原因不是“写得不熟”，而是缺少系统边界。

## 一、先定义设计令牌

先把颜色、间距、字体、圆角抽成基础令牌，再做组件层映射。
不要在组件里到处写硬编码值。

## 二、组件边界必须清晰

一个组件只负责自己的视觉与状态，不跨层改父级样式。
跨层覆盖越多，耦合越高，后续重构成本越大。

## 三、状态样式统一约定

至少统一这几类状态：

- default
- hover
- focus
- disabled
- error

状态规范统一后，交互体验会立刻变得一致。

## 四、主题能力预留但不滥用

很多团队一开始就做全量主题，结果开发成本飙升。
更实用的做法是先支持关键令牌切换，再逐步扩展。

## 五、重构顺序建议

1. 先收敛基础令牌
2. 再抽高复用组件（按钮、表单、卡片）
3. 最后清理历史页面样式

这样的收益是可见的：新需求上线更快，视觉也更稳。

## 结语

CSS 架构的目标不是“最优雅”，而是“可持续”。
当你把边界和约定建立起来，样式系统会从负担变成资产。
