# Nuxt3 + Tailwind 博客性能优化实践（首屏到交互）

很多个人博客会在功能做完后才开始关注性能，结果是“页面看起来没问题，但用户感觉慢”。
这篇文章给你一套**从可测量指标出发**的优化路径，适合技术博客场景。

## 一、先定义目标，不要盲目优化

先把目标写清楚，优化才不会跑偏：

- 首页 LCP < 2.5s（4G 网络）
- 文章页 TTFB 稳定，避免抖动
- 首屏请求数控制在可解释范围
- 移动端滚动与交互无明显卡顿

建议从 Lighthouse + Web Vitals 入手，先记录当前基线，再迭代。

## 二、页面策略：不是所有页面都要同一种渲染

Nuxt3 的优势是灵活，不要“一刀切” SSR。

- 首页、文章列表：SSR + 缓存（保证 SEO 和首屏）
- 文章详情：SSR + 短周期再验证
- 低频页面：可延迟加载或按需渲染

这样既保证可抓取，也能控制后端压力。

## 三、图片是博客性能的第一杀手

技术博客里的封面图、截图、流程图很多，图片优化收益最大。

建议策略：

1. 封面图固定比例，避免 CLS（布局抖动）
2. 首屏只加载必要尺寸，列表卡片不要直接上原图
3. 正文长图懒加载，减少首屏压力
4. 对图片做合理压缩与格式选择

如果你接了对象存储 + CDN，记得统一使用可控的图片参数策略，不要每次手写。

## 四、Tailwind 的性能关键在“约束”

Tailwind 本身不慢，慢的是“失控的样式规模”。

- 约束可用色板与间距体系
- 避免同一组件出现大量一次性类名组合
- 提取复用组件，降低模板复杂度
- 减少无意义动画和复杂阴影

目标不是“类名更少”，而是“样式系统更稳定”。

## 五、数据请求：避免首屏 waterfall

常见问题是页面先拿文章，再拿作者，再拿分类，形成串行请求。

优化方式：

- 后端返回前端需要的聚合数据
- 前端并发请求必要资源
- 用缓存减少重复查询
- 列表页和详情页拆分查询深度

一句话：把“多次等待”变成“有计划的一次等待”。

## 六、交互体验：让用户感知“快”

性能不仅是数字，也是感受。

- 首屏用骨架屏而不是长时间白屏
- 切页时保持布局稳定，避免闪烁
- 长文目录、代码块等重组件延迟加载
- 提供明确的加载状态和失败兜底

用户能感知到的连续性，往往比纯技术指标更重要。

## 七、可落地的优化顺序

如果你时间有限，按这个顺序做：

1. 图片策略（体积、尺寸、懒加载）
2. 页面渲染策略（SSR 与缓存）
3. 请求聚合与并发
4. Tailwind 体系收敛
5. 交互细节打磨

每一步都先测再改，避免“优化了很多但没收益”。

## 结语

博客站点的性能优化，不是一次性工程，而是一个持续过程。
把指标、策略、工程习惯统一起来，站点会越来越稳，内容体验也会更好。

如果你刚开始做 Nuxt3 + Tailwind 博客，最建议先从**图片与首屏策略**开始，它们往往是 ROI 最高的两件事。
