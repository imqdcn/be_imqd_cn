# Nuxt3 数据请求深挖：从能用到可维护

很多 Nuxt 项目早期都能跑起来，但一旦页面变多、内容变复杂，数据请求层很快失控：

- 页面里散落着大量重复请求
- 错误处理风格不一致
- 缓存策略靠“感觉”
- 一改接口，全站回归

这篇文章给你一个实用框架：把请求层当成“领域能力”来设计，而不是临时拼接。

## 一、先分层：页面不该直接关心所有细节

建议把数据请求拆成三层：

1. API Client 层：只做请求与基础错误映射
2. Repository 层：组织业务语义（文章列表、详情、专题聚合）
3. 页面层：只消费可展示的数据

这样页面就不会被 query 参数和容错细节污染。

## 二、并发优先，避免无意义串行

常见反模式是“先拿文章，再拿作者，再拿标签”。
如果这些资源在业务上没有前后依赖，应默认并发。

对博客站点来说，更好的方式是让后端返回前端需要的聚合数据，减少 N+1 请求。

## 三、缓存要有边界，不要全局一刀切

缓存不是越多越好，而是要区分数据新鲜度：

- 文章详情：短期缓存 + 再验证
- 分类/标签：更长缓存
- 热门榜单：可按分钟刷新

如果你把所有接口都“永久缓存”，用户看到的将是过期内容；
如果全都实时请求，服务端与客户端都会承压。

## 四、错误处理要可预期

请求失败不是异常情况，而是常态。
设计时至少覆盖：

- 网络错误
- 鉴权错误
- 资源不存在
- 接口超时

并且要有统一格式，页面才能做统一降级（重试按钮、兜底文案、错误页）。

## 五、把“可观测性”写进请求层

生产问题通常不是“有没有 bug”，而是“无法快速定位”。
建议在请求层记录：

- 请求耗时
- 请求参数摘要
- 失败分类
- 重试次数

当线上慢下来时，你才能知道是接口慢、序列化慢，还是渲染慢。

## 六、实操建议：博客站的最小可用方案

如果你现在要落地一版稳定方案，可以按这个顺序：

1. 抽 API Client（统一超时、错误映射）
2. 抽 Repository（文章、专题、知识库）
3. 页面改成仅消费 Repository
4. 针对列表/详情加分级缓存
5. 加基础日志与耗时统计

这套做完后，你会发现：性能不只是快了，代码也更敢改了。

## 结语

真正优秀的数据请求层，目标不是“写得炫”，而是“让业务长期可维护”。
在 Nuxt3 里，把请求设计成稳定能力，后续所有页面都会受益。
